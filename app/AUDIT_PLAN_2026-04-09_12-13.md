# Аудит VN-двигуна: Рефакторинг кирилиці + Баг-фікси

## Context

Кастомний HTML/JS двигун візуальної новели (Warframe: Drifter Loop, ~8700 рядків JS). Діалогова система — flat entries з умовами (як Hades/Disco Elysium), не branching tree.

**Проблема**: Кирилиця (українські імена персонажів) використовується як ключі/ідентифікатори в коді (gameState, дикти, масиви). Має бути ТІЛЬКИ для display output. Внутрішні ключі — short ID з CAST (ar, el, lt, am, ao, qu).

**Принцип**: Новий персонаж додається ВИКЛЮЧНО в таблицю персонажів (characters.csv / `c()` виклик). Все інше — автоматично з даних CAST.

### A2. `charFlag()` — `app/js/systems/combat/injuries.js:351`

Флаги використовують ПОВНІ латинські імена (arthur, eleanor, lettie...) — читабельно.
Додати поле `flag_name` в CAST metadata: `"arthur"`, `"eleanor"`, `"lettie"`, `"amir"`, `"aoi"`, `"quincy"`.
`charFlag()` читає з CAST замість hardcoded map:
```js
function charFlag(name) {
  if (CAST[name]) return CAST[name].flag_name || name;
  return name.toLowerCase();
}
```
Приймає short ID, повертає повне латинське ім'я для flags. Новий персонаж задає своє flag_name в таблиці.

## PHASE B: Баг-фікси

### B1. `checkForcedDialogue` ігнорує пріоритет

**Файл**: `app/js/systems/world/dispatcher.js:295-309`
**Баг**: Повертає ПЕРШИЙ matching entry замість найпріоритетнішого
**Фікс**: Зібрати всі eligible, відсортувати по priority, повернути найвищий

### B2. `day30_warning` залежить від позиції Артура

**Файл**: `app/js/scenes/finale.js:22-34`
**Баг**: Попередження про день 31 спрацьовує тільки якщо Артур в локації гравця
**Фікс**: Додати спеціальну перевірку в `showLocation()` або `_returnToLocation()` що не залежить від NPC presence

### B3. `_tryAutoAdvance` — leak інтервалів

**Файл**: `app/js/ui/quickmenu.js:150-163`
**Баг**: Кілька викликів під час typing створюють кілька setInterval
**Фікс**: Трекити interval ID, clearInterval перед створенням нового

### B4. `_refreshUI` працює на нон-game екранах

**Файл**: `app/js/systems/core/flags.js:31-36`
**Баг**: DOM маніпуляції при кожному setFlag/clearFlag навіть коли game-screen прихований
**Фікс**: Guard з перевіркою `game-screen.style.display !== "none"`

### B5. `checkAllFriends` пропускає персонажів без хімії

**Файл**: `app/js/scenes/finale.js:11-18`
**Баг**: Ітерує `gameState.chemistry.values` замість CAST — якщо з кимось ніколи не взаємодіяли, вони не враховуються
**Фікс**: Ітерувати CAST, перевіряти `(gameState.chemistry.values[short] || 0) < 120`

## PHASE C: Автоматизація для нових персонажів

Принцип: новий персонаж додається ТІЛЬКИ в таблицю, все інше автоматично.

### C1. Розширити CAST metadata

Додати поля до `c()` реєстрації / characters.csv:
- `gossip_targets` — масив short ID (кому розповідає плітки). `GOSSIP_ROUTES` генерується автоматично з CAST.
- `coffee_pref` — ID улюбленого напою. `COFFEE_PREFERENCES` генерується з CAST.
- `help_request` — `{message, location}` або null. `HELP_REQUEST_ENTRIES` генерується з CAST.

### C2. Генерувати daily flags динамічно з CAST

В `day.js` замінити hardcoded масив daily flags на:
```js
var dailyFlags = [];
for (var k in CAST) {
  var cf = charFlag(k);  // "arthur", "lettie" etc
  dailyFlags.push("helped_" + cf + "_today");
  dailyFlags.push("coffee_given_" + cf + "_today");
}
// Плюс загальні flags
dailyFlags.push("helped_someone_today");
```

### C3. Прибрати всі інші hardcoded списки персонажів

Замінити на ітерацію `Object.keys(CAST)` де можливо.

## Рішення

- **Внутрішні ключі** (gameState дикти, масиви): **short ID** (ar, el, lt, am, ao, qu)
- **Назви флагів**: **повні латинські імена** (dating_arthur, helped_lettie_today) — логічно і читабельно
- **Display output**: `charName()` → українські імена
- **Без конвертерів** — де потрібен ID, передається ID. Де потрібне ім'я — ім'я. charFlag() для flag names.
- **Storytelling data** (gossip routes, coffee prefs, help requests) → в CAST metadata
- **Daily flags** → генеруються динамічно з CAST
